# GUI Issue Playbook

## 목적

GUI 문제를 다시 분석할 때 같은 조사와 설명을 반복하지 않도록, 자주 나오는 문제 유형과 확인 순서를 정리한다.

## 공통 문제 유형

### 1. i18n 키가 그대로 노출됨

증상:

- `rigready_vectorizer_gui.start`
- 비슷한 번역 키 문자열이 버튼/헤더/설명에 그대로 표시됨

주요 원인:

- 앱 폴더 `locales.json`이 존재하지만 `main.py`에서 `load_extra_strings()`를 호출하지 않음
- 카테고리 `_engine/locales.json`에는 키가 없고 앱 전용 locale에만 정의돼 있음
- `t("some.key")` 호출에 fallback 문자열이 없어 키 문자열이 그대로 노출됨

표준 해결:

1. 앱 `main.py`에 앱 폴더 `locales.json` 로딩 추가
2. 핵심 헤더/버튼에는 `t("key", "Fallback")` 형태의 fallback 추가
3. 새 번역 키는 가능하면 카테고리 공용 `_engine/locales.json` 또는 앱 `locales.json`에 정리
4. 캡처 테스트로 키 문자열 노출이 사라졌는지 확인

### 2. 구형 또는 일관성 없는 레이아웃

증상:

- 다른 앱과 분위기가 다름
- 공통 헤더/푸터 구조를 따르지 않음
- 공통 테마 변수보다 자체 색상/배치를 많이 사용함

주요 원인:

- `BaseWindow`를 사용하지 않고 `ctk.CTk`를 직접 상속
- `place()`와 고정 geometry 중심 레이아웃
- 공통 컴포넌트보다 앱 내부 커스텀 프레임 조합 비중이 큼

표준 해결:

1. 가능하면 `BaseWindow` 기반으로 정렬
2. 공통 헤더, 푸터, 진행 상태, 버튼 패턴 재사용
3. 고정 절대 좌표보다 `grid`/`pack` + weight 기반 반응형 정렬 사용
4. 색상과 버튼 스타일은 공통 theme 상수 우선 사용

### 3. 버튼/텍스트가 잘리거나 일부만 보임

증상:

- 상단 모델 관리자 텍스트 일부만 보임
- 버튼 라벨이 잘림
- 한 줄에 여러 컨트롤이 몰려 폭 부족 발생

주요 원인:

- 좁은 윈도우 폭에 `pack(side="left")`로 많은 위젯을 한 줄 배치
- 공용 프레임 내부 최소 폭/줄바꿈 정책 없음
- 라벨, 상태 텍스트, 액션 버튼이 같은 헤더 행을 공유

표준 해결:

1. 한 줄 헤더를 `grid` 2열 이상으로 분리
2. 상태 카드와 액션 버튼을 세로 분리하거나 다음 줄로 내림
3. 최소 창 폭과 카드 최소 폭을 같이 설계
4. 캡처 테스트에서 100% 스케일과 더 좁은 폭 둘 다 확인

### 4. 하드코딩 문자열/영문 UI 혼용

증상:

- 일부는 번역 키, 일부는 영문/한글 하드코딩
- 툴팁/에러 메시지/파일 대화상자 제목 언어가 섞임

주요 원인:

- 점진 이식 과정에서 `t()` 적용이 일부만 됨
- 예외 처리나 파일 대화상자 제목은 나중에 추가되어 누락

표준 해결:

1. 사용자 노출 문자열을 모두 `t()` 경유로 수집
2. `messagebox`, `filedialog`, 버튼, placeholder, tooltip까지 포함해 검사
3. 하드코딩 문자열은 마지막 단계가 아니라 초기에 정리

### 5. 하단 액션 버튼이 화면 밖으로 밀림

증상:

- 실행/취소 버튼이 보이지 않음
- 진행 바와 상태 라벨이 캡처에서 사라짐
- 긴 본문 영역 아래에 액션이 묻혀 실제 사용이 어려움

주요 원인:

- 긴 본문과 하단 버튼을 모두 `self.main_frame`에 쌓아 둠
- `BaseWindow.footer_frame`를 사용하지 않아 푸터 고정이 깨짐
- 창 높이가 부족한데 `scrollable=True` 또는 스크롤 가능한 본문 구성이 없음

표준 해결:

1. 본문은 `self.main_frame`, 실행/취소/진행 상태는 `self.footer_frame`으로 분리
2. 설정 항목이 긴 앱은 `BaseWindow(..., scrollable=True)` 사용
3. `FileListFrame`, 로그 박스, 모델 관리자 카드 높이를 줄여 첫 화면에서 액션이 보이게 조정
4. 캡처 결과에서 버튼 2개와 진행 바가 한 화면에 같이 보이는지 확인

Flet 표준 해결:

1. `window_profile`을 앱 유형에 맞게 재선정
2. 본문과 하단 실행 영역을 분리하고 `action_bar()` 사용
3. 툴바/보조 버튼은 `wrap=True`와 최소폭 규칙 적용
4. 그래도 밀리면 본문만 스크롤되도록 바꾸고 액션바는 고정

### 6. 앱 래퍼가 앱 루트 대신 카테고리 루트를 참조함

증상:

- 앱 전용 `locales.json`이 있어도 로딩되지 않음
- 앱 아이콘, 부가 리소스, 템플릿, 예제 파일이 앱 폴더 기준으로 열리지 않음

주요 원인:

- 래퍼 `main.py`에서 `APP_ROOT = Path(__file__).resolve().parents[1]`처럼 잘못 계산
- `CTX_APP_ROOT`가 카테고리 루트를 가리켜 `BaseWindow`의 locale 탐색이 틀어짐

표준 해결:

1. 래퍼에서는 `APP_ROOT = Path(__file__).resolve().parent`를 사용
2. `LEGACY_ROOT = APP_ROOT.parent / "_engine"` 형태로 앱 루트와 엔진 루트를 분리
3. `CTX_APP_ROOT`를 앱 폴더로 유지한 상태에서 캡처 테스트 재검증

## 분석 순서

1. 캡처 PNG에서 문제 유형을 먼저 분류
2. 앱 `main.py`가 `locales.json`을 로드하는지 확인
3. GUI 구현이 `BaseWindow`인지 직접 `ctk.CTk`인지 확인
4. 상단 헤더, 액션 버튼, 카드 내부가 `pack(side="left")` 과밀 구조인지 확인
5. 하드코딩 문자열과 `t()` fallback 누락 위치 찾기
6. 푸터 액션이 `footer_frame`이 아닌 본문 안에 묻혀 있지 않은지 확인
7. 수정 후 GUI 캡처 재실행

## 수정 후 체크리스트

1. 번역 키가 그대로 남지 않았는가
2. 버튼 라벨과 상태 텍스트가 잘리지 않는가
3. 상단 헤더에서 모델 카드/액션 버튼이 온전히 보이는가
4. 다크 테마와 공통 카드 스타일이 다른 앱과 어색하게 충돌하지 않는가
5. 실행/취소 버튼과 진행 바가 첫 화면에 노출되는가
6. stderr 없이 캡처가 끝나는가
