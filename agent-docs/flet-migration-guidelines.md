# Flet Migration Guidelines

## 목적

이 문서는 `Contexthub-Apps` 저장소의 기존 Python GUI 앱을 `customtkinter`/Tk 계열에서 `Flet` 기반 UI로 점진적으로 이관할 때, 다른 AI 에이전트가 같은 기준으로 판단하고 작업하도록 돕기 위한 공통 가이드다.

대상은 특정 앱 하나가 아니라, 이 저장소의 Python GUI 앱 전반이다.

다른 세션에서 Flet 작업을 다시 시작할 때의 메인 문서는 이 파일이다.

## 이 문서가 필요한 이유

- 현재 일부 Python GUI 앱은 레이아웃 유연성, 스레드 처리, 상태 관리, 복잡한 상호작용에서 유지보수 비용이 커지고 있다.
- 앱별로 제각각 재작성하면 카테고리마다 구조와 UX가 갈라진다.
- 단순히 “Flet로 바꾸자” 수준으로 접근하면 기존 컨텍스트 메뉴, 패키징, 공통 런타임, `_engine` 구조를 깨뜨릴 위험이 있다.

즉, Flet 전환은 “UI 프레임워크 교체”가 아니라 다음 네 가지를 동시에 맞춰야 하는 작업이다.

- 앱 진입 구조 유지
- 공통 런타임과의 연결 유지
- 기존 기능 회귀 방지
- 카테고리별 일관된 UX 확보

## 기본 원칙

### 1. 한 번에 전부 갈아엎지 않는다

- 전체 카테고리를 한 번에 Flet로 옮기지 않는다.
- 먼저 앱 단위로 전환한다.
- 더 안전한 순서는 다음과 같다.
  1. 복잡한 추론 없이 UI 상호작용이 많은 앱
  2. 레이아웃 문제와 상태 관리 비용이 큰 앱
  3. AI 추론/외부 프로세스 연동 앱

### 2. UI와 기능 로직을 분리한다

- Flet로 옮길 때 가장 먼저 해야 할 일은 기존 GUI 코드에서 “실제 기능 로직”을 떼어내는 것이다.
- 추론, 파일 처리, 모델 호출, 외부 프로세스 호출은 Flet 위젯 코드 안에 직접 넣지 않는다.
- Flet 화면은 상태 표시와 이벤트 연결만 담당하고, 실제 작업은 기존 `_engine` 서비스/헬퍼 함수로 위임한다.

### 3. 기존 앱 ID, manifest, 실행 진입점은 가능하면 유지한다

- 앱의 서비스 정체성은 `manifest.json`과 앱 폴더 구조에 있다.
- Flet 전환 때문에 앱 ID를 바꾸지 않는다.
- 컨텍스트 메뉴, Quick Launcher, manual 경로도 가능하면 유지한다.

### 4. Flet 전환의 목표는 “더 보기 좋게”가 아니라 “더 안정적이고 관리 가능하게”다

- 단순한 시각 변경만을 위해 대규모 전환을 하지 않는다.
- 아래 조건 중 2개 이상이면 Flet 전환 우선순위가 높다.
  - 창 레이아웃이 자주 깨진다
  - 동적 상태가 많아 Tk 위젯 갱신 비용이 크다
  - 채팅형/카드형/리스트형 인터랙션이 많다
  - 모바일/웹 유사 UX가 더 자연스럽다
  - 비동기 작업 상태 표시가 복잡하다

### 5. Tk/CTk는 최종 상태로 남기지 않는다

- 이 저장소의 Python GUI 최종 목표는 Tk/CTk와 Flet의 공존이 아니라 Flet 단일화다.
- 새 기능은 Tk/CTk 구현에 추가하지 않는다.
- wrapper가 이미 Flet 진입을 사용하면 Tk/CTk 구현은 레거시 제거 후보로 본다.
- Tk/CTk 파일을 남겨야 한다면 이유와 제거 조건을 문서에 남긴다.

## 전환 대상 선정 기준

### Flet 전환에 잘 맞는 앱

- 채팅형 인터페이스
- 카드 리스트/대시보드형 앱
- 멀티 스텝 워크플로 앱
- 진행률/작업 큐/로그 패널이 중요한 앱
- 다양한 조건부 UI가 필요한 앱

예:

- `ai/qwen3_tts`
- `ai_light/prompt_master`
- `document/doc_scan`
- 여러 입력/출력 상태를 가진 `image`, `video`, `audio` 계열 툴

### Flet 전환 우선순위가 낮은 앱

- 단일 버튼 + 파일 선택 수준의 매우 단순한 앱
- 창을 거의 쓰지 않고 외부 툴 실행만 하는 앱
- GUI보다 추론/배치 처리 비중이 압도적으로 큰 앱

이런 앱은 기존 Tk 구조를 유지해도 비용 대비 이득이 작을 수 있다.

## 권장 마이그레이션 전략

### 전략 A. 앱별 점진 이관

가장 권장되는 방식이다.

1. 기존 앱의 기능 로직을 서비스 함수로 추출
2. 기존 `main.py` 진입 구조는 유지
3. GUI 구현만 Flet 화면으로 교체
4. 캡처 테스트와 실제 기능 테스트로 회귀 확인

장점:

- 리스크가 작다
- 패키징 구조를 거의 건드리지 않는다
- 실패 시 원복 범위가 작다

추가 원칙:

- 단, 전략 A를 쓰더라도 “legacy Tk 유지”를 장기 상태로 두지 않는다.
- 1차 포팅이 끝났으면 다음 단계는 Tk 경로 제거 여부 판단이다.

### 전략 B. 카테고리 공용 Flet 셸 도입

앱 수가 많고 비슷한 화면 구조가 반복될 때 고려한다.

예:

- 공통 상단 상태 바
- 공통 로그 패널
- 공통 작업 큐 카드
- 공통 파일 입력/출력 섹션

장점:

- 디자인 일관성 확보
- 새 앱 제작 속도 상승

단점:

- 공용 추상화가 너무 빨리 커지면 다시 `_engine` 복잡성이 증가
- 초기 설계가 잘못되면 여러 앱이 한 번에 묶여서 수정 비용이 커짐

권장:

- 처음부터 큰 공용 셸을 만들지 말고, 2~3개 앱 이관 후 공통 패턴이 실제로 반복될 때만 추출

## 템플릿 우선 원칙

반복되는 공통 오류가 2번 이상 나오면, 다음 앱부터는 즉흥 구현보다 템플릿을 먼저 제시하는 편이 맞다.

현재 저장소 기준:

- 새 Flet 포팅 시작점은 `agent-docs/templates/flet-porting-template.md`

이 문서는 다음을 강제한다.

- `main.py` wrapper 역할 고정
- `service / state / flet_app` 분리
- 공통 토큰 사용
- 미구현 핸들러와 placeholder 상태 노출 금지

규칙:

- 새 Flet 앱을 만들거나 기존 GUI를 Flet으로 옮길 때, 먼저 이 템플릿과 현재 가이드를 함께 읽는다.
- 템플릿에서 벗어나는 구조를 쓰려면 이유가 분명해야 한다.

## 폴더/코드 구조 권장안

### 유지할 것

- 앱 폴더: `{category}/{app_id}/`
- 카테고리 공용 엔진: `{category}/_engine/`
- 앱 진입: `{category}/{app_id}/main.py`
- 매뉴얼: `{category}/{app_id}/manual.md`

### 권장 분리

앱이 Flet로 이관될 때 다음 구조를 권장한다.

```text
{category}/_engine/features/.../{app_name}/
  service.py
  state.py
  flet_app.py
  adapters.py
```

- `service.py`
  - 실제 기능 실행
  - 파일 처리, 추론, 외부 프로세스 호출
- `state.py`
  - 화면 상태 구조
  - 현재 작업, 선택 항목, 로그, 에러 상태
- `flet_app.py`
  - Flet 페이지 구성
  - 컨트롤 이벤트 연결
- `adapters.py`
  - 기존 런타임이나 레거시 helper와의 연결

중요:

- Flet UI 파일 안에서 직접 모델 호출/긴 작업을 하지 않는다.
- legacy Tk 파일 안에만 남아 있는 서비스 로직이 없도록 분리한다.

## 런타임 연결 원칙

### 1. main.py는 얇게 유지

`main.py`는 다음 역할만 해야 한다.

- 앱 루트/환경 변수 설정
- `_engine` 경로 진입
- Flet 앱 실행 함수 호출
- 컨텍스트 메뉴 입력 경로 전달

즉, 기존 wrapper형 `main.py` 철학을 유지한다.

### 2. 공통 환경 변수와 런타임 가정은 유지

현재 저장소는 다음과 같은 공통 가정을 가진다.

- `CTX_APP_ROOT`
- `CTX_SHARED_ROOT`
- 카테고리 `_engine` 경로
- `ai` 카테고리 Conda 우선 실행 규칙

추가 규칙:

- wrapper `main.py`는 가능하면 `LEGACY_ROOT`, `SHARED_ROOT`, `SHARED_PACKAGE_ROOT`를 모두 `sys.path`에 주입한다.
- Flet 앱이 아니더라도, 이후 공용 토큰이나 i18n 유틸을 재사용할 가능성이 있으면 shared runtime 경로를 미리 맞춰 둔다.

Flet 전환 시에도 이 계약을 깨지 않는다.

### 3. 외부 프로세스는 반드시 비동기/백그라운드로 처리

Flet 화면에서 다음은 메인 UI 흐름을 막지 않게 처리한다.

- AI 추론
- ffmpeg/외부 바이너리 호출
- 큰 파일 변환
- 긴 배치 처리

권장:

- 작업 큐 객체
- 백그라운드 스레드 또는 subprocess
- 진행 상태만 UI로 전달

## 상태 관리 가이드

Flet에서는 “컨트롤 직접 수정”보다 “상태 구조 -> UI 반영”이 더 중요하다.

권장 상태 분류:

- `app_state`
  - 현재 모드
  - 선택 파일/출력 폴더
  - 글로벌 설정
- `view_state`
  - 열려 있는 패널
  - 선택된 카드/버블
  - 현재 탭
- `job_state`
  - idle/running/done/error
  - progress text
  - progress ratio
  - outputs
- `diagnostic_state`
  - warnings
  - logs
  - last exception

중요:

- 상태는 dict보다 명시적 구조를 권장한다.
- 최소한 함수 경계를 넘는 상태는 일관된 키를 유지한다.

## UX 규칙

### 공통 Flet 계층을 먼저 만든다

Flet 앱이 2개 이상 생기기 시작하면, 아래 항목은 앱마다 하드코딩하지 말고 공유 런타임으로 끌어올린다.

- 창 기본 크기, 최소 크기
- 컬러 토큰
- spacing 토큰
- radius 토큰
- 공통 dialog/modal shell
- 공통 side panel / surface 카드 스타일
- 상태 배지 색 규칙

현재 저장소 기준 권장 위치:

```text
dev-tools/runtime/Shared/contexthub/ui/flet/
  tokens.py
  theme.py
  window.py
  dialogs.py
  layout.py
```

실제 허브 런타임에도 같은 구조를 유지한다.

중요:

- 새 Flet 앱은 이 계층이 생긴 뒤에는 page 색상, 창 크기, 라운드 값, 패딩 값을 직접 숫자로 박아 넣지 않는다.
- 앱 전용 시각 실험이 필요하더라도, 공용 토큰을 override 하는 방식으로 처리한다.
- 하단 실행 영역, 주요 버튼 최소폭, wrap 가능한 툴바도 공용 `layout.py` helper를 우선 사용한다.

### 공통 화면 구조

복잡한 앱은 가능하면 아래 순서를 따른다.

1. 상단 상태 바
2. 메인 작업 영역
3. 보조 패널
4. 하단 실행/결과 영역

예:

- 상단 상태 바:
  - 모델/환경 상태
  - 현재 모드
  - 경고 요약
- 메인 작업 영역:
  - 채팅/카드/리스트/미리보기
- 보조 패널:
  - 프로필
  - 고급 설정
  - 로그
- 하단:
  - 실행 버튼
  - 진행률
  - 최근 결과

### 조건부 UI는 접어둔 상태가 기본

- 초보자 경로와 고급자 경로를 분리한다.
- 자주 쓰는 핵심만 먼저 보인다.
- 드문 설정은 `More`, `Advanced`, `Details`로 접는다.

### destructive action은 기본 노출하지 않는다

- 삭제
- 재설정
- 캐시 삭제
- 모델 재다운로드

이런 액션은 `More` 또는 별도 확인 흐름으로 둔다.

### 결과 경험은 “생성 후 바로 이해 가능”해야 한다

결과 카드에는 최소한 다음이 있어야 한다.

- 생성 타입
- 출력 파일
- 재생/열기
- 재실행
- 실패 시 에러 요약

AI 오디오 앱은 추가로:

- waveform 또는 길이
- 참조 오디오 상태
- 생성 방식(`Preset Voice`, `Clone from Audio`, `Design New Voice`)

## 창 / 모달 / 레이아웃 중앙화 규칙

### 창 크기 규칙

- 데스크톱 기본 폭/높이는 공유 `window.py`에서만 관리한다.
- 앱 코드에서 `page.window_width = 1280` 같은 하드코딩을 반복하지 않는다.
- 최소 크기도 반드시 같이 정의한다.
- 앱은 숫자 대신 `window_profile`을 선언한다.

권장:

- `compact`: 단일 작업/짧은 폼
- `form`: 일반 입력 폼 + 하단 액션
- `two_pane`: 좌우 2단 구조
- `wide_canvas`: 큰 미리보기/캔버스 중심
- `table_heavy`: 채널/리스트/테이블 중심

규칙:

- 앱이 특별히 더 커야 하면 새 숫자를 앱 코드에 직접 쓰지 말고, `window.py`에 named preset을 추가한다.
- 창 프로필을 바꿔도 버튼 잘림이 남으면 먼저 프로필을 더 키우지 말고 `본문 스크롤 + 하단 액션바 고정 + 툴바 wrap`을 우선 검토한다.

### 액션바 / 버튼 폭 규칙

- 하단 실행 영역은 공통 `action_bar()`로 묶는다.
- 상태 텍스트는 좌측, 주요 CTA는 우측을 기본으로 한다.
- 보조 버튼은 같은 행에서 `wrap=True`가 되도록 둔다.
- 주요 CTA는 최소폭 `180`, 툴바 버튼은 최소폭 `148`, 작은 보조 버튼은 `120` 전후를 기본으로 한다.

이유:

- 캡처 환경, DPI, 번역 길이 차이로 인해 버튼 폭 부족이 반복되기 때문이다.
- 창 크기만 키우는 방식보다 최소폭 규칙과 공통 액션바가 재발 방지에 더 효과적이다.

### 팝업 / 다이얼로그 규칙

- `AlertDialog` 내용 레이아웃, padding, radius, 버튼 행은 공용 helper를 사용한다.
- destructive dialog, progress dialog, confirm dialog를 앱마다 다시 짜지 않는다.
- 모달은 폭을 고정하고, 텍스트 길이에 따라 세로만 늘어나게 둔다.

권장:

- 일반 confirm: 폭 `420~460`
- progress dialog: 취소 버튼은 우하단 고정
- destructive action: 버튼 순서는 `Cancel -> Confirm`

### 컬러 중앙화 규칙

- 앱 배경, 카드 배경, 입력창 배경, border, 강조색, 상태색은 전부 `tokens.py`에서 읽는다.
- 모드 색(`custom_voice`, `voice_clone`, `voice_design`)과 상태 색(`ready`, `queued`, `done`, `error`)도 공용 토큰으로 관리한다.
- 프로필 색상도 앱마다 즉흥 랜덤이 아니라, 공용 seed 함수로 계산한다.

금지:

- `#11161F`, `#253042` 같은 값을 앱 코드 전반에 반복 하드코딩

### spacing / grid 규칙

- `8 / 12 / 16 / 20 / 24` 같은 spacing scale을 공용으로 고정한다.
- 카드 내부 패딩, section 간 간격, panel spacing은 이 토큰만 쓴다.
- 화면이 커져도 spacing을 앱마다 제멋대로 늘리지 않는다.

권장:

- input row: `12~16`
- section gap: `16~20`
- dialog padding: `24`

### radius 규칙

- 작은 필드, 카드, 큰 surface, pill 배지는 서로 다른 radius token을 쓴다.
- radius도 공용 scale로 통일한다.

권장:

- field: `12~16`
- card: `16~20`
- conversation bubble: `24`
- pill badge: `999`

### 그리드 규칙

- 좌우 2단 구조 앱은 `메인 영역 + 보조 패널` 패턴을 우선한다.
- 보조 패널 폭도 공용 named width를 사용한다.
- 카드 리스트는 spacing과 최대 폭을 일정하게 유지하고, 무의미한 자유 배치를 피한다.

권장:

- side panel: `320~360`
- narrow detail tray: `280~320`
- 메인 리스트는 `expand=True` 유지

## 성능 가이드

### Flet이어도 성능이 자동으로 좋아지지 않는다

- 너무 잦은 전체 페이지 갱신

### 권장

- 고빈도 진행률은 텍스트/비율 수준으로 제한
- 로그는 버퍼링하거나 최근 N개만 유지
- 이미지/파형/미리보기는 축약 렌더링

### 피해야 할 것

- 실시간 애니메이션을 무분별하게 추가
- 초당 수십 번 UI 전체 refresh
- 대용량 파일 파싱을 페이지 이벤트 안에서 직접 처리

## 레거시 제거 기준

Tk/CTk 구현을 삭제하거나 보류할지 판단할 때 아래를 순서대로 본다.

1. wrapper가 Tk 경로를 더 이상 호출하지 않는가
2. `manual.md`나 사용자 문서에 Tk fallback이 남아 있지 않은가
3. locale/helper/service가 Tk 파일에만 갇혀 있지 않은가
4. 캡처/스모크/수동 검증이 Flet 경로 기준으로 끝났는가
5. 실제 기능상 미구현 흐름이 Flet에 남아 있지 않은가

위 조건을 만족하면 Tk/CTk 파일은 기본적으로 정리 대상으로 본다.

## 의존성 게이팅 규칙

Flet 포팅 앱은 "부팅 성공"과 "실제 기능 성공" 사이에 의존성 리스크가 자주 남는다.

이번 세션에서 실제로 확인된 사례:

- `merge_to_exr`: `cv2` 미설치 시 실행 버튼 이후 실패
- `rigreader_vectorizer`: `vtracer` 미설치 시 벡터화 단계 실패

권장 규칙:

1. `service.py`에 `get_missing_dependencies()`를 둔다.
2. `flet_app.py`의 실행 버튼 핸들러에서 먼저 프리체크한다.
3. 누락이 있으면 작업 시작 전에 상태 라벨에 즉시 표시한다.
4. 선택 의존성(예: `psd_tools`)과 필수 의존성(예: `vtracer`)을 구분한다.

즉, "백그라운드 스레드에서 실패 로그 확인"이 아니라 "클릭 즉시 사용자 안내"를 기본으로 한다.

## 잘못된 선택 방지 규칙

다른 에이전트가 실제 전환 중 자주 실수할 수 있는 지점을 명시적으로 막는다.

### 1. UI 교체와 백엔드 재설계를 동시에 하지 않는다

잘못된 선택:

- Flet로 옮기는 김에 추론 호출 방식, 모델 구조, 출력 포맷까지 한 번에 바꾸기

권장:

- 먼저 기존 백엔드 계약을 그대로 유지
- UI는 그 계약을 호출하는 얇은 셸만 교체

### 2. 공유 런타임 helper에 과하게 의존하지 않는다

잘못된 선택:

- 새 Flet 앱이 기존 shared helper의 내부 경로 가정에 깊게 묶이기

문제:

- 로컬 테스트, 패키징 경로, 실제 허브 런타임이 미세하게 다를 때 쉽게 깨진다

권장:

- 앱 전용 service에서 필요한 실행 계약을 한 번 더 감싼다
- 특히 AI 추론 subprocess 호출은 앱 전용 launcher/service로 감싼다

### 3. Flet 전환 첫 단계에서 완전한 기능 패리티를 강요하지 않는다

잘못된 선택:

- 첫 이관 버전에서 모든 미세 UX, 애니메이션, 고급 위젯까지 한 번에 맞추기

권장:

- 1차 목표는 구조와 기능 회귀
- 2차 목표는 UX polish

### 4. Flet 의존성 미설치 상태를 무시하지 않는다

잘못된 선택:

- Flet 미설치 환경에서 바로 앱이 죽도록 전환

권장:

- 앱을 실제 서비스 단계로 올릴 생각이라면, 초기에라도 Flet 설치 경로를 먼저 확정한다.
- `requirements.txt`, 실제 런타임 Python, 로컬 테스트 Python이 모두 같은 방향으로 맞는지 확인한다.

추가 규칙:

- fallback은 임시 실험 단계에서만 제한적으로 허용된다.
- 패키징 대상 앱, 사용자 테스트 대상 앱, 매뉴얼에 노출된 앱은 fallback 없이 단일 Flet 진입으로 정리한다.
- 즉 `fallback으로 문제를 숨긴 채 배포`하는 선택은 금지에 가깝다.

### 5. 모드/상태가 많은 앱에서 상태 구조 없이 바로 위젯부터 짜지 않는다

잘못된 선택:

- 메시지 선택, 프로필 편집, 작업 진행, 결과 표시를 모두 이벤트 핸들러 안에서 즉흥적으로 관리

권장:

- `app_state`, `view_state`, `job_state`를 먼저 분리
- 상태 전이 이름을 명확히 둔다

### 6. 미완성 핸들러를 연결한 채 포팅 완료로 간주하지 않는다

잘못된 선택:

- 버튼은 보이지만 `pass` 상태
- `on_pick_files`, `on_clear_all`, `btn_extract` 같은 이름은 UI에 연결했지만 실제 정의가 없음
- TODO가 핵심 기능 경로에 남아 있음

권장:

- 미구현 흐름이 있으면 버튼을 숨기거나 명시적으로 disabled 처리
- 대표 기능 경로에는 `pass`를 남기지 않는다

### 7. wrapper와 import 계약을 섞지 않는다

잘못된 선택:

- `flet_app.py`나 `service.py`는 package-relative import를 쓰는데
- `main.py`는 top-level script처럼 직접 돌림

문제:

- 런타임에서 `attempted relative import with no known parent package`

권장:

- package import로 통일하거나
- top-level import 규칙으로 전체를 통일한다

한 앱 안에서 두 계약을 섞지 않는다

### 8. 실패 처리에서 원래 예외를 가리지 않는다

잘못된 선택:

- `except ImportError as e:` 뒤에서 선언되지 않은 `logger`를 써서 다른 예외로 덮어쓰기

권장:

- 가장 단순한 실패 메시지로 유지
- import 실패는 import 실패로 보이게 남긴다

## Qwen3-TTS 기준 권장 참조 패턴

`qwen3_tts` 같은 앱은 다음 구성을 참조 패턴으로 본다.

```text
qwen3_tts.py
  기존 추론 백엔드 유지

qwen3_tts_service.py
  프로필 로드/저장
  clone 상태 진단
  백엔드 subprocess 실행

qwen3_tts_state.py
  메시지/앱 상태 구조

qwen3_tts_flet_app.py
  Flet 화면 구성
  이벤트 -> service 호출 연결

main.py
  Flet 전용 실행
```

이 패턴의 장점:

- 기능 로직을 재사용할 수 있다
- UI 실험을 반복해도 추론 백엔드는 안정적으로 유지된다
- 다른 AI 앱으로 복제하기 쉽다

## 권장 결론

이 저장소에서 Flet 전환은 “전면 재작성”보다 “앱별 점진 이관 + 공통 패턴 검증 후 재사용”이 가장 안전하다.

즉, 권장 접근은 다음 한 줄로 요약된다.

- 먼저 기능 로직을 분리하고, 그 다음 UI만 Flet로 교체하며, 공통 패턴은 2~3개 앱에서 검증된 뒤에만 추출한다.
